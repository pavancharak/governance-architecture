ChangeSet Template

Formal Governance Amendment

1. ChangeSet Identity

ChangeSet ID:
(Unique identifier, e.g. CS-0001)

Change Type:
ADD | UPDATE | DEPRECATE

Date:
(YYYY-MM-DD)

Proposed By:
(Human identity)

2. Target

Target Document:
(e.g. ontology.md)

Target Section:
(e.g. Section 2.2 — Definitional Entities)

Target Record ID (if applicable):
(Hash or name)

3. Current Definition

Paste the exact current text that is being changed.

This section must match the canonical document exactly.

4. Proposed Definition

Paste the full proposed new text.

Do not describe the change — write the full replacement.

5. Rationale

Explain why this change is needed.

This must be human reasoning, not AI reasoning.

6. Invariants Preserved

List which axioms and global invariants remain valid.

Example: - Determinism - Human Authority - Non-Retroactivity - Closed Ontology

7. Risks Introduced

Describe any new risks, ambiguities, or edge cases.

If none, explicitly state:
“No new risks identified.”

8. Backward Compatibility

Does this change affect existing historical records?

Yes | No

If Yes, explain why this does not violate non-retroactivity.

9. Impact Analysis

Describe what components are affected: - Engine - Registry - Schemas - Processes - Humans

10. Approval

Human Approvals Required:

  • Name:
    Signature:
    Date:

  • Name:
    Signature:
    Date:

11. Activation

Effective Version:
(e.g. architecture_v2)

Activation Condition:
(Immediate | Scheduled | Conditional)

12. Post-Activation Notes

Optional notes after deployment.

This section is filled only after activation.
